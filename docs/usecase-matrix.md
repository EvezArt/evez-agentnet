# Use case matrix for the governed runtime

This matrix explains where the cognition-governed path is meaningfully different from simpler agent stacks.

| Use case | What most systems do | What `evez-agentnet` does differently | Why it matters |
|---|---|---|---|
| Fast content loop | Rank signals and immediately generate/ship | Inserts daemon arbitration and can refuse to ship under unresolved mass or entropy | Reduces premature action and brittle monetization |
| Forecasting under ambiguity | Pick top forecast and suppress rivals | Preserves rival futures and tracks predictor entropy | Better for unstable domains where one confident guess is dangerous |
| Autonomous building | Wait for external opportunity ranking | Allows daemon build queue to create artifact pressure from within the system | Supports self-directed development rather than pure reaction |
| Multi-agent orchestration | Route between named agents | Maintains a visible authority chain through lineage, checkpoint, and ship governance | Makes responsibility legible |
| Long-running runtime | Restart and lose cognitive state | Reloads checkpointed branch state, unresolved residue, and identity mode | Preserves continuity across rounds |
| Audit / trust | Show logs after the fact | Emits proof, receipt, dashboard, and spine digest surfaces | Makes inspection a first-class feature |
| RSI / speculative planning | Keep hypotheses in reports or logs | Pushes RSI back in as branch injections | Lets speculation affect later cognition instead of dying as commentary |

---

## Simple decision rule

Use the governed path when any of these are true:

- you care about uncertainty more than output volume
- you need audit artifacts by default
- you want autonomous build pressure, not only reactive ranking
- you need shipping to be refused sometimes for principled reasons
- you want runtime continuity instead of session amnesia

Use the legacy path only when you want the fastest flat pipeline and do not care about governed cognition.
